## Purpose

The purpose of this project is to expose several private local LLM machines as
one OpenAI-compatible model provider behind Cloudflare AI Gateway.

The intended user experience is simple:

```text
Coding agent or app
  -> Cloudflare AI Gateway
  -> one custom provider
  -> one dynamic route or model name
  -> best available private local machine
  -> local llama.cpp or compatible inference server
```

The user should not need to know which machine is serving a request. A coding
agent should call one AI Gateway URL, while the router decides whether the
request should go to the office RTX 3090, a Mac Studio, a Linux workstation, or
another enrolled machine.

## Product Goal

Build a private inference mesh with these properties:

- One AI Gateway custom provider endpoint for all local inference nodes.
- Multiple Windows, macOS, and Linux nodes.
- Nodes reachable privately through Cloudflare One Client, WARP, Mesh, and
  Workers VPC.
- A Cloudflare Worker router that receives OpenAI-compatible requests.
- D1 persistence for router config, setup state, node records, model profiles,
  sessions, and audit data.
- A Durable Object scheduler for serialized in-flight reservations and hot
  routing decisions.
- A local node agent with a browser UI for setup and observability.
- One-line node installation generated by the router admin UI.
- GitHub Actions for Worker deploys and node-agent releases.

The first useful version should route AI Gateway traffic to one WARP-enrolled
machine running an existing local llama.cpp server. Multi-node scheduling and
fully managed model downloads come after that path is proven.

## Design Principles

### Keep The Public Surface Small

Only the Worker is public. Local nodes do not expose public hostnames and do not
need inbound port forwarding. The Worker enforces application-level bearer
tokens for AI Gateway, node registration, and node upstream calls.

### Separate Control Plane From Data Plane

Inference traffic is the data plane:

```text
AI Gateway -> Worker -> selected node -> local runtime
```

Registration, setup, health, model profiles, and admin screens are the control
plane:

```text
Admin UI / node agent -> Worker -> Durable Object
```

Separating these concepts avoids leaking admin credentials into inference
calls and keeps routing logic understandable.

### Prefer Private Network Routing Over Public Per-Node URLs

The preferred transport is Workers VPC to Cloudflare Mesh. Each node registers
its private Mesh IP and port. The Worker builds the target URL itself. Nodes do
not register arbitrary URLs.

This reduces public exposure and avoids managing one Cloudflare Tunnel hostname
per machine.

### Preserve Session Affinity

Long-context coding work is expensive to prefill. If a session jumps between
machines, each node has to rebuild its cache. The router should prefer sending
all requests for a coding session to the same node, even if another node is
currently idle.

### Make The Node Agent Opinionated

The node agent should not ask users for Cloudflare account IDs, Durable Object
names, model flags, or AI Gateway details. Those belong to the router and
model profiles. A node should need only:

```text
router URL
setup token
optional node name
```

### Start Narrow, Then Generalize

Do not build the full desktop product before proving the Cloudflare transport.
The first milestone is:

```text
AI Gateway -> Worker -> Workers VPC Mesh fetch -> one WARP client -> llama.cpp
```

## Architecture Summary

### Target Architecture

```text
Client / coding agent
  -> Cloudflare AI Gateway
  -> AI Gateway custom provider
  -> Cloudflare Worker router
  -> D1 persistent state + Durable Object scheduler
  -> Workers VPC binding
  -> Cloudflare Mesh
  -> WARP-enrolled node
  -> node agent
  -> local llama.cpp / Ollama / LM Studio / vLLM
```

### Logical Planes

```text
Provider plane
  AI Gateway custom provider and dynamic route.

Router plane
  Worker HTTP API, auth checks, request forwarding, streaming.

State plane
  D1 durable state plus Durable Object scheduling and reservations.

Node plane
  Cross-platform agent, local UI, runtime supervision, model management.

Release plane
  GitHub Actions, Worker deploys, agent builds, installer artifacts.
```

### Data Plane Request Path

```text
1. Client calls AI Gateway dynamic route.
2. AI Gateway calls custom provider URL on the Worker.
3. Worker verifies the provider bearer token.
4. Worker parses the OpenAI-compatible request.
5. Worker maps the public model name to the active internal model profile.
6. Worker asks Durable Object for a node reservation.
7. Durable Object selects an eligible node.
8. Worker sends the request through Workers VPC to the node Mesh IP.
9. Node agent validates the upstream token.
10. Node agent forwards to local inference runtime.
11. Runtime response streams back through the same path.
12. Worker releases the reservation when the stream ends or fails.
```

### Control Plane Node Path

```text
1. Admin creates a short-lived setup token in Worker setup UI.
2. Admin copies a one-line install command.
3. Installer starts the node agent.
4. Node agent claims setup token.
5. Worker returns permanent node credentials and model profile.
6. Node agent detects Mesh IP and starts heartbeat.
7. Durable Object stores the live node lease.
```

## Component Responsibilities

### Cloudflare AI Gateway

Purpose:

- Public AI gateway used by clients and coding agents.
- Logging, dynamic routing, provider abstraction, and optional BYOK/provider key
  storage.

Responsibilities:

- Receive client requests.
- Route selected model or dynamic route to the inference mesh custom provider.
- Send the provider bearer token when calling the Worker, if configured through
  BYOK/provider key storage.
- Apply configured timeout and retry settings.

AI Gateway should not:

- Know about individual local machines.
- Hold node registration tokens.
- Dispatch directly to WARP client IPs.
- Know which concrete local model is currently active behind a stable router
  alias.

### Cloudflare Worker Router

Purpose:

- Act as the OpenAI-compatible custom provider endpoint.
- Hide local node topology from AI Gateway.
- Enforce authentication and routing policy.

Responsibilities:

- Serve `/v1/models` and `/v1/chat/completions`.
- Verify `ROUTER_PROVIDER_TOKEN` on provider API requests.
- Forward chat completions to selected nodes through Workers VPC.
- Stream node responses back to AI Gateway.
- Serve first-run setup UI on the `workers.dev` URL.
- Optionally provision a custom domain after first-run configuration.
- Create or update the AI Gateway custom provider and dynamic route.
- Serve install scripts.
- Accept node claim and heartbeat requests.

The Worker should stay mostly stateless. Routing state belongs in the Durable
Object.

### D1 Database

Purpose:

- Store all durable router state and operational history.

Responsibilities:

- Store router setup state.
- Store Cloudflare resource IDs created during setup.
- Store admin config.
- Store node records and durable node credentials.
- Store public model aliases and model profiles.
- Store setup tokens and token hashes.
- Store session mappings.
- Store request/audit history if retained.

D1 is the source of truth for persistent state. A Worker or Durable Object can
rebuild live scheduler state from D1 if an isolate is restarted.

References:

- D1:
  <https://developers.cloudflare.com/d1/>
- D1 Worker bindings:
  <https://developers.cloudflare.com/d1/worker-api/>

### Durable Object Scheduler

Purpose:

- Provide one strongly coordinated place for live node state and scheduling.

Responsibilities:

- Cache live node records and leases from D1.
- Track in-flight reservations.
- Track sessions and sticky node mappings.
- Apply model alias mapping.
- Score eligible nodes.
- Return 429 when capacity is exhausted.
- Expire stale nodes.

Why Durable Objects:

- The scheduler needs consistent in-flight counts.
- Two Worker isolates must not reserve the same single-GPU node at the same
  time.
- Durable Objects provide per-object serialized execution, which fits routing
  reservations and short-lived hot state.
- D1 remains the durable source of truth.

References:

- Durable Objects:
  <https://developers.cloudflare.com/durable-objects/>
- Durable Object storage:
  <https://developers.cloudflare.com/durable-objects/api/storage-api/>
- Durable Object alarms:
  <https://developers.cloudflare.com/durable-objects/api/alarms/>

### Workers VPC And Cloudflare Mesh

Purpose:

- Let the Worker call private WARP-enrolled node IPs without public node
  hostnames.

Expected Worker binding:

```jsonc
{
  "vpc_networks": [
    {
      "binding": "MESH",
      "network_id": "cf1:network",
      "remote": true
    }
  ]
}
```

Expected Worker call:

```ts
const response = await env.MESH.fetch(
  "http://100.96.12.34:11434/v1/chat/completions",
  requestInit
);
```

Important operating assumptions:

- The node must be enrolled in Cloudflare Zero Trust through Cloudflare One
  Client, not only consumer WARP.
- Mesh or device IPs must be enabled.
- The local node listener must bind to the Mesh IP or another address reachable
  from the WARP interface.
- The local firewall must permit inbound traffic on the node API port from the
  Mesh path.
- The Worker builds URLs from validated `meshIp` and `port`. The node never
  registers a full URL.

### Node Agent

Purpose:

- Make each local machine self-contained and observable.
- Shield the router from OS-specific details.
- Manage local runtime lifecycle in a consistent way.

Responsibilities:

- Run as a background service on Windows, macOS, and Linux.
- Serve a local UI on `http://127.0.0.1:17777`.
- Detect WARP status and Mesh IP.
- Claim setup tokens.
- Store permanent node credentials.
- Start and stop the inference engine.
- Download and verify model files in later phases.
- Enforce local bearer-token auth for Worker-to-node calls.
- Report metrics and heartbeat to the Worker.

The node agent should not require the user to configure Cloudflare API details.
It receives its router URL, credentials, and model profile from the setup flow.

## End-To-End Flows

### First Deployment Flow

Intent:

The first deployment should require only GitHub repository secrets and should
bring up a Worker on its `workers.dev` hostname. The Worker can then configure
itself through a first-run admin screen.

Flow:

```text
1. User adds GitHub secrets.
2. GitHub Actions deploys Worker to workers.dev.
3. User opens the Worker setup URL.
4. User authenticates with first-run setup token.
5. User optionally configures custom domain.
6. User selects Cloudflare AI Gateway.
7. User enters route name, for example freestyler.
8. Worker creates custom provider pointing to itself.
9. Worker creates dynamic route using that provider.
10. Worker creates provider bearer token and displays it once.
11. User adds that token as the AI Gateway provider key or BYOK value.
12. Worker generates node install command.
```

Why this way:

- It avoids requiring Cloudflare Access on day one.
- It avoids manual dashboard-only setup for custom providers and dynamic routes.
- It keeps the Worker usable before a custom domain exists.
- It gives a clean split between bootstrap token and long-lived provider token.

### Node Install Flow

Linux and macOS command:

```bash
curl -fsSL https://inference-mesh-router.example.workers.dev/install.sh | sudo IM_SETUP_TOKEN=abc123 sh
```

Windows PowerShell command:

```powershell
$env:IM_SETUP_TOKEN="abc123"; irm https://inference-mesh-router.example.workers.dev/install.ps1 | iex
```

Flow:

```text
1. Installer downloads the correct node-agent artifact.
2. Installer verifies checksum.
3. Installer installs the service.
4. Installer writes minimal bootstrap config.
5. Installer starts the local agent UI.
6. Agent calls POST /node/claim with setup token.
7. Worker returns permanent node token, upstream token, and model profile.
8. Agent detects Mesh IP.
9. Agent starts heartbeat.
10. Agent appears online in router admin UI.
```

Why this way:

- Install scripts carry no permanent secret.
- Setup tokens are short-lived and single-use.
- Local machines do not need Cloudflare API tokens.
- The node can be rebuilt or rotated without touching AI Gateway config.

### Inference Flow

Flow:

```text
1. Client sends OpenAI-compatible request to AI Gateway.
2. AI Gateway resolves dynamic route to custom provider.
3. AI Gateway calls Worker `/v1/chat/completions`.
4. Worker validates provider token.
5. Worker maps requested public model name to active internal profile.
6. Worker asks Durable Object for a reservation.
7. Durable Object selects a node.
8. Worker calls node agent through `env.MESH.fetch`.
9. Node agent validates upstream token.
10. Node agent forwards to local runtime.
11. Response streams back to AI Gateway.
12. Worker releases reservation.
```

Why this way:

- AI Gateway remains the only endpoint clients need.
- Durable Object prevents overbooking a single-GPU node.
- Worker-to-node token prevents arbitrary callers on the Mesh from using the
  runtime.
- Streaming preserves OpenAI-compatible behavior.

## Security And Trust Model

### Trust Boundaries

The system has four different trust boundaries. They must use separate
credentials.

```text
Client -> AI Gateway
  AI Gateway auth or client auth, depending on the user's Gateway policy.

AI Gateway -> Worker
  ROUTER_PROVIDER_TOKEN.

Node agent -> Worker
  Per-node token issued after setup-token claim.

Worker -> node agent
  NODE_UPSTREAM_TOKEN or per-node upstream token.
```

### Token Classes

```text
ROUTER_PROVIDER_TOKEN
  Stored in AI Gateway as provider key or BYOK value.
  Used only by AI Gateway when calling Worker `/v1/*`.

INIT_SETUP_TOKEN
  Initial bootstrap token used to open first-run setup.
  Stored as GitHub secret or generated once by deploy workflow.

ONE_TIME_SETUP_TOKEN
  Short-lived token generated by admin UI for a node install.
  Used once by `/node/claim`.

NODE_TOKEN
  Permanent per-node bearer token.
  Used by node heartbeats and unregister calls.

NODE_UPSTREAM_TOKEN
  Token Worker sends when calling node local API.
  Can be global for MVP and per-node later.

CLOUDFLARE_API_TOKEN_RUNTIME
  Stored only as Worker secret.
  Used by setup UI to provision Cloudflare resources.
```

### Why Not One Token

One shared token would be easier, but it creates bad blast radius:

- A compromised node could impersonate AI Gateway.
- A leaked provider token could register fake nodes.
- A copied install command could enroll unlimited machines.
- A local runtime token leak could grant admin access.

Separate tokens let each breach be contained and rotated independently.

### Public Routes And Auth

Recommended route behavior:

```text
GET /
GET /setup
  First-run setup UI.
  Requires INIT_SETUP_TOKEN until setup is complete.

GET /admin/*
POST /admin/*
  Requires admin session or ADMIN_TOKEN.
  Cloudflare Access is optional later, not required for MVP.

GET /install.sh
GET /install.ps1
  Public read is acceptable.
  Scripts contain no permanent secret.

POST /node/claim
  Public route.
  Requires ONE_TIME_SETUP_TOKEN.

POST /node/heartbeat
POST /node/unregister
  Public route.
  Requires NODE_TOKEN.

GET /v1/models
POST /v1/chat/completions
  Public route from Cloudflare perspective.
  Requires ROUTER_PROVIDER_TOKEN.
```

### Cloudflare Access Position

Cloudflare Access is useful for human admin UI later. It should not be required
for the first version because:

- AI Gateway would need to send Access service-token headers.
- Node agents would also need Access service-token headers.
- Access does not replace app-level node identity.

First version:

```text
workers.dev setup UI
  protected by first-run token and later admin token/session

custom domain
  optional, configured from setup UI

Access
  optional hardening step after core path works
```

### Request Handling Rules

The Worker should route requests in this order:

```text
1. Health checks.
2. Static setup/admin UI assets.
3. Installer scripts.
4. Node claim and heartbeat endpoints.
5. Admin endpoints.
6. Provider auth for `/v1/*`.
7. OpenAI-compatible provider endpoints.
8. 404.
```

Do not put provider auth in front of `/node/*`, `/install.*`, or setup UI.
Those routes use different auth.

### Chat Completion Handling

Required behavior:

```text
1. Read JSON body.
2. Validate request body size.
3. Normalize `model`.
4. Extract session ID.
5. Reserve node in Durable Object.
6. Rewrite model to node upstream model name.
7. Forward to node through `env.MESH.fetch`.
8. Stream upstream body.
9. Release reservation on stream end, error, or missing body.
10. Add observability headers.
```

Required headers to node:

```http
Authorization: Bearer <NODE_UPSTREAM_TOKEN>
Content-Type: application/json
X-Inference-Mesh-Request-Id: <request_id>
X-Inference-Mesh-Session: <session_id>
```

Do not forward these headers to nodes:

```text
client Authorization
Cloudflare API token
admin token
node registration token
AI Gateway metadata that contains secrets
```

### Streaming Rules

For `stream: true`, pass through Server-Sent Events unchanged:

```text
node SSE stream -> Worker body -> AI Gateway -> client
```

For long context, prefill can take many seconds before the first token. The
plan should prefer:

- `stream: true` for coding agents.
- AI Gateway timeout of at least 60 seconds, higher if supported.
- AI Gateway retries set to 0 or 1 for local long-running routes.
- Worker retries only before generation starts.
- No keepalive bytes for non-stream JSON responses.

SSE comment keepalives can be tested later, but only if AI Gateway and the
client tolerate them.

### Language And UI Choice

Use Go for the agent.

Why:

- One binary per OS and architecture.
- Practical Windows service support.

- The agent is primarily a service.

- A browser UI avoids Electron, Tauri, Wails, or platform GUI installers in the

### Agent Runtime Responsibilities

The agent should:

- Run as a background service.
- Serve local UI and API on `127.0.0.1`.
- Detect Cloudflare One Client and Mesh IP.
- Claim setup token.

- Heartbeat to Worker.
- Listen for Worker-to-node inference requests on Mesh-reachable address.
- Enforce `Authorization: Bearer <NODE_UPSTREAM_TOKEN>`.
- Proxy OpenAI-compatible requests to local runtime.
- Start and stop runtime process.
- Report runtime state and metrics.
- Drain before shutdown.
- Check for new signed node-agent releases.
- Download and stage verified updates.
- Offer update and restart through the local UI.

The node agent should support self-update, but updates must be explicit,


### Update Check Flow

The agent should periodically check:

```text

```

It should also allow manual checks from the local UI:

```text

```

The update checker should:

1. Compare semantic versions.
2. Ignore prereleases unless the node is on a prerelease channel.
3. Select the asset matching OS and architecture.
4. Find `checksums.txt`.
5. Find signature or provenance files when available.
6. Report available update in the dashboard.

GitHub documents release metadata through the Releases REST API and binary
downloads through release assets. Release assets can be downloaded from

- GitHub Release Assets API:
  <https://docs.github.com/rest/releases/assets>

### Update Download And Verification

The agent should download to a staging directory:

```text
Windows:
  C:\ProgramData\InferenceMesh\updates\<version>\

Linux:
  /var/lib/inference-mesh/updates/<version>/

macOS:
  /Library/Application Support/InferenceMesh/updates/<version>/
```

Verification steps:

1. Download release archive.
2. Download `checksums.txt`.
3. Verify archive SHA256 against `checksums.txt`.
4. Verify signed checksum file when signing is configured.
5. Unpack into staging directory.
6. Run `inference-mesh-agent version` from staged binary.
7. Mark update as ready.

The release workflow should sign checksum files. GoReleaser supports generating
checksums and signing checksum artifacts, which is enough for the agent to
verify that the downloaded archive matches a signed release manifest.

Reference:

- GoReleaser signing:
  <https://goreleaser.com/customization/sign/>

## Model Runtime Strategy

### Default Engine: llama.cpp

Use `llama-server` as the default engine.

Why:

- Cross-platform.
- Supports CUDA, Metal, Vulkan, ROCm, and CPU builds depending on platform.
- Exposes OpenAI-compatible endpoints.
- Gives direct control over long-context flags.
- Allows tuning context length, flash attention, KV cache quantization, GPU
  layers, aliases, and reasoning format when supported.

Reference:

- llama.cpp server:
  <https://github.com/ggml-org/llama.cpp/tree/master/tools/server>

### Model Profiles

The router owns model profiles. The agent consumes them. A model profile is the
internal runtime definition for one concrete model on one hardware class.

External clients and AI Gateway should not depend on these concrete names.
They should call stable public aliases such as:

```text
mesh-default
freestyler
code-worker
```

The router maps those aliases to active profiles.

Why Gemma 4 26B-A4B is in scope:

- Google documents Gemma 4 as open weights with long-context support up to
  256K tokens.
- The 26B-A4B variant is a consumer-GPU-sized MoE candidate.
- It is a useful competitor to Qwen3.6-27B for coding-agent quality, latency,
  and reasoning behavior.
- It can be served through `llama-server` as an OpenAI-compatible endpoint when
  using a compatible GGUF build.

Gemma caveats:

- Use a recent llama.cpp build because Gemma 4 compatibility has changed over
  time.
- Test text-only first. Multimodal support is not required for the coding-agent
  route and may need additional projector/runtime handling.
- Treat 256K as a validation target, not a guarantee. Runtime overhead and KV
  cache behavior decide whether the 3090 profile is stable.

Why profiles live in the router:

- New installs get updated defaults without rebuilding the agent.
- Different hardware classes can get different profiles.
- Admin can control which models are allowed.
- AI Gateway route names can map cleanly to model aliases.
- Public AI Gateway names can stay stable while the active internal model
  changes from Qwen to Gemma or back.

### Initial Runtime Scope

Phase 1 does not need automatic model download. It can use an existing local
llama.cpp server. This proves networking and AI Gateway behavior first.

Phase 4 adds:

- llama.cpp binary management.
- model download.
- checksum verification.
- process supervisor.
- one-click Start and Stop.

## Model Virtualization And Stable Gateway Names

### Intent

AI Gateway should be configured once and then left alone. The router should own
model choice, model rollout, and node-agent profile propagation.

The external contract should be stable:

```text
AI Gateway route: freestyler
AI Gateway provider: custom-inference-mesh
AI Gateway provider model: mesh-default
Worker public model: mesh-default
```

The internal implementation can change:

```text
mesh-default -> qwen36-27b-256k-3090
mesh-default -> gemma4-26b-a4b-256k-3090
mesh-default -> small-smoke-test-32k
```

This lets the administrator switch between Qwen, Gemma, smaller smoke-test
models, or hardware-specific profiles without reconfiguring AI Gateway.

### Public Model Alias

The Worker should expose public models from router config:

```json
{
  "publicModels": [
    {
      "name": "mesh-default",
      "description": "Default local inference mesh model",
      "activeProfileId": "gemma4-26b-a4b-256k-3090",
      "fallbackProfileIds": [
        "qwen36-27b-256k-3090"
      ]
    }
  ]
}
```

`GET /v1/models` should return public aliases, not every internal runtime name.

Example response:

```json
{
  "object": "list",
  "data": [
    {
      "id": "mesh-default",
      "object": "model",
      "owned_by": "inference-mesh"
    }
  ]
}
```

### Request Rewriting

Inbound request from AI Gateway:

```json
{
  "model": "mesh-default",
  "messages": [],
  "stream": true
}
```

Router lookup:

```text
mesh-default
  -> activeProfileId: gemma4-26b-a4b-256k-3090
  -> internal model name: gemma4:26b-a4b
```

Outbound request to selected node:

```json
{
  "model": "gemma4:26b-a4b",
  "messages": [],
  "stream": true
}
```

The AI Gateway route stays configured with `mesh-default`. Only the Worker
router config changes.

### Profile Versioning

Model profiles need a version so node agents can detect changes:

```json
{
  "profileId": "gemma4-26b-a4b-256k-3090",
  "profileVersion": 3,
  "model": "gemma4:26b-a4b",
  "context": 262144
}
```

```text

```

Node heartbeat should include:

```json
{
  "nodeId": "win3090-office",
  "readyProfiles": [
    {
      "profileId": "qwen36-27b-256k-3090",
      "profileVersion": 1,
      "loaded": true
    }
  ],
  "activeRuntimeProfile": "qwen36-27b-256k-3090"
}
```

```json

    {
      "profileId": "gemma4-26b-a4b-256k-3090",
      "profileVersion": 3,
      "action": "prepare"
    }
  ],

    {
      "publicModel": "mesh-default",
      "activeProfileId": "gemma4-26b-a4b-256k-3090"
    }
  ]
}
```

### Rollout Flow

Safe model swap:

```text
1. Admin changes `mesh-default` desired profile to Gemma 4.
2. Router increments profile version.
3. Node agents receive desired profile on heartbeat.
4. Nodes download or prepare Gemma 4.
5. Nodes start runtime or warm model.
6. Nodes heartbeat `readyProfiles` including Gemma 4.
7. Router switches `mesh-default` active profile when enough nodes are ready.
8. New requests route to Gemma 4.
9. Old Qwen profile remains available for rollback.
```

For MVP, the rollout can be manual:

```text
admin changes active profile
node prepares profile
admin switches active alias when node is ready
```

Later, the router can support canary and automatic readiness gates.

### Compatibility With AI Gateway

This architecture is compatible with AI Gateway because AI Gateway only sees a
normal OpenAI-compatible provider:

```text
POST /v1/chat/completions
model: mesh-default
```

The Worker performs all internal model selection and rewrites the model name
before forwarding to the node. AI Gateway does not need to know whether the
node is running Qwen, Gemma, or another local runtime.

### Compatibility With OpenAI APIs

The first supported endpoint should be:

```text
POST /v1/chat/completions
```

The Worker can add these later:

```text
POST /v1/responses
POST /v1/completions
POST /v1/embeddings
```

Do not make `/v1/completions` the primary target for Gemma 4. Gemma 4 is an
instruction/chat model, and the router should normalize coding agents onto
chat completions first.

### Why This Is Better Than Reconfiguring AI Gateway

Stable public aliases give:

- one permanent AI Gateway custom provider
- one permanent dynamic route
- one permanent client configuration
- router-controlled model swaps
- fast rollback
- node-agent propagation through heartbeat responses

The cost is that the Worker must own alias mapping, profile versioning, and
request rewriting. That is the right place for this logic because the Worker
already owns scheduling and node selection.

### Workflow Design

The workflow design should follow the same discipline as Codeflare:

- PR checks are the normal merge gate.
- Deploy is manual, not automatic on every push.
- Production deploys are restricted to `main`.
- Deploy repeats the critical checks before changing Cloudflare state.
- Node-agent release artifacts are created by the deploy workflow.
- Actions are pinned to commit SHAs.
- Permissions are minimal and set per workflow or per job.
- Concurrency groups prevent overlapping deploys or duplicate expensive checks.
- Security workflows are separate from deploys.

Why node-agent releases belong in deploy:

The node agent has a self-update mechanism. That mechanism checks GitHub
Releases and downloads signed artifacts. Therefore every real deployment that
should update the fleet must also produce a release artifact set. If the Worker
is deployed without publishing compatible agent artifacts, the router can
advertise desired behavior that existing agents cannot run.

## Failure Modes

### Node Stops Heartbeating

Behavior:

- Lease expires.
- Node is removed from eligible routing.
- Existing session mapping can remain briefly.
- If node does not return, next request can reroute or return 429.

### Node Is Busy

Behavior:

- If session affinity is not required, try another compatible node.
- If the request belongs to a hot sticky session, prefer 429 over moving it.

Reason:

Moving a long-context coding session can force another huge prefill and make
latency worse.

### Node Crashes Mid-Stream

Behavior:

- Release reservation.
- Increment recent failure count.
- Apply failure penalty.
- Return upstream stream failure or 502.
- Do not automatically retry after generation has started.

### WARP Disconnects

Behavior:

- Node heartbeat fails or reports disconnected.
- Lease expires.
- Router stops selecting node.

### AI Gateway Retries

Risk:

AI Gateway can retry a request and create duplicate local generations.

Mitigation:

- Set retries to 0 or 1 for long local routes.
- Let Worker retry only before generation starts.

### Context Cache Thrash

Risk:

Requests for one coding session bounce between nodes.

Mitigation:

- Explicit session ID.
- Sticky session mapping in Durable Object.
- Prefer busy response over moving a hot session.

## Alternatives Considered

### Per-Node Cloudflare Tunnel Hostnames

Alternative:

```text
Worker -> https://node-a.example.com
Worker -> https://node-b.example.com
```

Pros:

- Mature Cloudflare Tunnel path.
- Easier to test with existing Nginx Proxy Manager or cloudflared setup.
- No dependency on Workers VPC to Mesh behavior.

Cons:

- One public hostname per node.
- More DNS and Tunnel configuration.
- More Access/service-token complexity.
- Larger public attack surface.

Decision:

Use as fallback only. Prefer Workers VPC and Mesh for primary design.

### AI Gateway Directly To Each Node

Alternative:

```text
AI Gateway dynamic route -> per-node custom providers
```

Pros:

- Less custom router code.

Cons:

- AI Gateway does not know local node health or Mesh IPs.
- Hard to enforce session affinity.
- Hard to reserve single-GPU capacity.
- Node credentials and URLs leak into provider config.

Decision:

Reject. Keep one Worker router.

### Concrete Model Names In AI Gateway

Alternative:

```text
AI Gateway route model: qwen3.6:27b
AI Gateway route model: gemma4:26b-a4b
```

Pros:

- Direct and easy to understand.
- Less alias mapping logic in the Worker.

Cons:

- Every model change requires AI Gateway route edits.
- Client configuration can drift from router state.
- Rollback requires Cloudflare route changes instead of one router config
  change.
- Node-agent desired state cannot be coordinated cleanly with Gateway config.

Decision:

Reject for the main product. Use stable public aliases such as `mesh-default`
or `freestyler` in AI Gateway, then let the Worker map those aliases to active
internal profiles.

### D1 Only, KV, Or R2 Instead Of Durable Object

Pros:

- Simple persistent storage.
- Easy inspection.

Cons:

- Scheduler needs serialized reservation logic.
- KV eventual consistency is wrong for in-flight counts.
- D1 is the durable source of truth, but it should not be the only mechanism
  for hot single-node reservation arbitration.
- R2 is object storage, not scheduler state.

Decision:

Use D1 for all durable state and data persistence. Keep a Durable Object for
serialized scheduling, hot leases, and in-flight reservations.

### Cloudflare Access As Primary Auth

Pros:

- Strong human login story.
- Good admin protection.

Cons:

- AI Gateway and node agents need Access service-token headers.
- Access does not replace node identity.
- Adds friction to the first working version.

Decision:

Use app-level bearer tokens first. Add Access to admin UI later if desired.

### Native Desktop App

Options:

```text
Tauri
Wails
Electron
```

Pros:

- More native desktop feel.

Cons:

- Extra packaging complexity.
- Poor fit for headless Linux.
- Not necessary for a service dashboard.

Decision:

Use Go service with embedded local web UI.

### Ollama Or LM Studio As Default Engine

Pros:

- Easier manual setup.
- Familiar UI in LM Studio.
- Ollama has convenient CLI model management.

Cons:

- Less control over long-context flags.
- Harder to reproduce exact performance settings.
- More awkward for one-button managed service behavior.

Decision:

Use llama.cpp as default. Add adapters later.

### vLLM As Default Engine

Pros:

- Strong throughput on Linux NVIDIA.
- Production-grade OpenAI-compatible server.

Cons:

- Not cleanly cross-platform.
- GPU and CUDA expectations make Windows and macOS harder.

Decision:

Keep vLLM as optional Linux backend, not default.

### Public GitHub Releases Versus Worker Download Broker

Public release artifacts:

- Simple installers.
- Easy checksum verification.
- Source repo can remain private only if releases/artifacts are public enough.

Worker download broker:

- Keeps artifacts private.
- Requires Worker to serve binaries from R2 or GitHub with server-side token.

Decision:

Keep the repository private until the first Worker path is working. Before
shipping node-agent install and self-update, make the repository public and use
public GitHub Release artifacts plus checksums/signatures. Do not build an R2
or Worker download broker for v1.

## Finalized Decisions

These are fixed for the first implementation. There are no open architecture
questions remaining before implementation starts.

1. Repository and artifact distribution:
   Keep the repository private until the first Worker path works. Then make the
   repository public and use public GitHub Release artifacts for install and
   self-update.

2. Model source schema:
   Use `hfSpecifier`, `localFilename`, optional `sha256`, and
   `llamaServerModelArg` for Hugging Face model sources. Do not rely on
   ambiguous `repo` plus `file` fields.

3. State persistence:
   Use D1 for all durable state and data persistence. Keep a Durable Object for
   serialized scheduling, hot leases, and in-flight reservations.

4. BYOK/provider key:
   Instruct the user to manually add the generated bearer token as the AI
   Gateway provider key/BYOK value. Do not automate BYOK in v1.

5. Admin auth after first setup:
   Use `ADMIN_TOKEN` and an admin session for the MVP. Cloudflare Access is an
   optional hardening step after the custom domain exists; it is not required
   for the first working version.

6. Runtime token model:
   Use one `NODE_UPSTREAM_TOKEN` for MVP to prove the mesh and proxy path, then
   add per-node upstream tokens in production hardening.

7. Model profile defaults:
   `mesh-default` initially targets `qwen36-27b-256k-3090` for serious
   coding-agent validation. `gemma4-26b-a4b-256k-3090` is the fallback and
   benchmark profile. `small-smoke-test-32k` is available for fast validation.

8. Dynamic route automation:
   The first version creates or updates the custom provider, dynamic route,
   route version, and route deployment automatically. BYOK/provider-key entry
   remains manual.

9. Node listener binding:
   The agent prefers binding to the detected Mesh IP. If that fails, it may
   bind to `0.0.0.0` only with strict upstream-token auth and a visible firewall
   warning in local and admin status.

10. Hardware metrics:
    Start with best-effort platform probes such as `nvidia-smi`. Missing GPU
    metrics are reported as absent, not fabricated. Native GPU libraries can be
    added after the core path is stable.
