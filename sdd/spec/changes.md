## 2026-07-07

- Fixed input (prompt-prefix) caching never engaging, so every request re-prefilled the whole prompt (`prompt_tokens_details.cached_tokens` stayed 0 on a large shared prefix). The prefix cache that populates `cached_tokens` is a distinct MeshLLM mechanism from parallel lanes and has two gates codeflare never set. First, it must be enabled: without it MeshLLM defers to per-family auto-detection. Second, and load-bearing, the cache `payload_mode` must match the model: MeshLLM's Auto inference matches the `qwen3` substring and picks `resident-kv`, which silently no-ops for recurrent-hybrid architectures (`qwen35`, `qwen3.6`, `qwen3-next`, `falcon-h1`), so those must pin `kv-recurrent`. Codeflare now renders `[models.model_fit.prefix_cache]` with `enabled`, `payload_mode`, `max_entries` (capped at 128; the fallback overruns the KV cell pool and returns HTTP 502), and `shared_stride_tokens` / `shared_record_limit`, exposed as the `prefixCache` model tunable over `POST /admin/profiles/config` and `POST /api/v1/models/{id}`. A new custom model enables the cache, sets `parallel` to 4 (an omitted value let MeshLLM collapse to a single lane; Auto never planned 4 lanes as previously documented), and auto-selects `payload_mode` from the model reference (recurrent-hybrid families to `kv-recurrent`, dense families left Auto). With no cache hits every request held a lane through a full 13-46s prefill, so MeshLLM admission-rejected concurrent requests, which the gateway surfaced as `503 Model execution failed`; restoring cache hits frees the lanes. ([REQ-RUN-002](runtime-profiles.md#req-run-002-default-model-profiles), [REQ-RUN-003](runtime-profiles.md#req-run-003-managed-meshllm-runtime))
- Hardened node-agent mesh formation and restart supervision. The agent now auto-provisions the iroh mesh-peer transport port (the profile's UDP `--bind-port`) for inbound WARP traffic, at startup and again on every profile switch since the bind port moves with the selected model, so a default-deny host firewall no longer silently drops the QUIC stage handshake and strands a multi-node split at zero peers (previously only the TCP data-plane port was opened). Separately, a runtime restart is now bounded by a timeout so a `Stop` hung on a mesh-llm ignoring SIGTERM cannot strand the restart-pending latch and permanently suppress every future restart on that node until a manual relaunch. ([REQ-NODE-010](node-agent.md#req-node-010-inbound-mesh-firewall-provisioning), [REQ-RUN-010](runtime-profiles.md#req-run-010-meshllm-process-lifecycle))
- Added an on-demand Node Force Reload so an operator can restart a node's mesh-llm runtime from the Manage drawer or the API (`POST /admin/nodes/{id}/reload` and the automation twin `POST /api/v1/nodes/{id}/reload`), recovering a wedged runtime without SSH. The request stamps a one-shot nonce that rides the node's heartbeat response; the node drains and restarts mesh-llm exactly once, then echoes the nonce back so the router retires the directive, so it never loops nor re-fires after an agent restart. ([REQ-ADM-032](setup-admin.md#req-adm-032-node-force-reload), [REQ-NODE-012](node-agent.md#req-node-012-on-demand-runtime-reload), [REQ-API-004](control-plane-api.md#req-api-004-programmatic-node-management))
- Fixed the console Deactivate/Activate control doing nothing when clicked: the Manage drawer rendered the button but the client action dispatcher had no branch for it, so a click never reached `POST /admin/nodes/{id}/deactivate` or `/activate` (the endpoint, persistence, projection, and rendering were all correct, so the taint had been reachable only over the automation API). The dispatcher now posts to the deactivate/activate endpoint and refreshes status. ([REQ-ADM-030](setup-admin.md#req-adm-030-node-deactivation-and-activation))
- Closed the automation-API parity gaps so every operational console action has an authenticated `/api/v1` twin: added `POST /api/v1/mesh/rotate` (mesh secret rotation) and `GET` / `PUT /api/v1/settings` (the offline-prune window), and covered the already-shipped `POST /api/v1/nodes/{id}/reload` twin with an automation test. Each twin shares the console handler's validation and audit core so the two surfaces cannot diverge. ([REQ-SEC-006](security.md#req-sec-006-mesh-token-lifecycle), [REQ-ADM-020](setup-admin.md#req-adm-020-node-status-clarity-and-filtering), [REQ-ADM-032](setup-admin.md#req-adm-032-node-force-reload))
- The node agent now captures mesh-llm's own stderr and rides the most recent error-looking line (an OOM, a CUDA error, a lane that never came up) plus the console's raw `node_state` on the heartbeat metrics as `runtimeDetail` and `nodeState`, so the reason a runtime is wedged is legible from the control plane instead of only the host journal. ([REQ-OBS-011](observability.md#req-obs-011-runtime-error-surface))

([REQ-ADM-027](setup-admin.md#req-adm-027-model-naming-and-rename), [REQ-ADM-021](setup-admin.md#req-adm-021-model-serving-configuration))
- Fixed the node agent SIGTERM-looping its `mesh-llm` child on roughly every 15s heartbeat, which kept a served model from ever staying up. Three restart paths each forced a relaunch every tick: the profile reconcile never marked the profile loaded after the asynchronous `Start`, so it restarted on every heartbeat; a `create` bootstrap directive was treated as an unconditional restart even for the already-running creator; and the router told the elected seed to `join` its own mesh once its invite token landed in the token set, flipping the seed's role every tick. The elected seed now always receives `create` even after its own token lands and only non-seed peers receive `join`, and the pre-restart drain now waits on the MeshLLM console `inflight_requests` as well as the local proxy counter so a restart never lands while a response is still generating. ([REQ-RUN-010](runtime-profiles.md#req-run-010-meshllm-process-lifecycle), [REQ-RUN-006](runtime-profiles.md#req-run-006-private-mesh-formation), [REQ-RUN-008](runtime-profiles.md#req-run-008-router-mesh-membership-authority))
- Fixed a second mesh restart loop where an expired seed left a stale token island: when a seed was elected but went silent, the router used to clear only the seed id and keep the leftover mesh id and tokens, so it never re-elected and kept handing the surviving node its own invite token as a `join` target, pointing it at a mesh only it could see and restarting it every heartbeat. The router now clears the whole rotation (seed, mesh id, and tokens) when a seed expires, never returns a node its own token (join uses only other nodes' tokens), and re-elects a node whose own token is the only one left so it creates a fresh mesh. The node agent also clears its tracked mesh id and invite token on each new process launch and mirrors the console mesh id (clearing it when the console reports none), so a stale id can no longer be reported or trip the mesh-id-diff restart. ([REQ-RUN-008](runtime-profiles.md#req-run-008-router-mesh-membership-authority), [REQ-RUN-006](runtime-profiles.md#req-run-006-private-mesh-formation))
